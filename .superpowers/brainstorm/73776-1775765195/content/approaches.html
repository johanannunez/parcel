Metrics & Financials: Architecture Approach

How should the dashboard, payouts, and property detail pages get their data?

A

Server Components + Parallel Queries (Recommended)

All three pages stay as async Server Components. Each page fans out its Supabase queries with Promise.all and renders server-side. The time period switcher uses URL search params (?period=ytd) so the selected period is shareable and bookmarkable.

Pros

  • Matches every existing portal page pattern
  • Zero client-side JS for data fetching
  • Period selection works with browser back/forward
  • SEO-friendly (not relevant here, but free)

Cons

  • Full page reload on period switch (mitigated by Next.js streaming)
  • No animated transitions between periods
B

Hybrid: Server Component Shell + Client Period Switcher

The page shell and initial data load are Server Components. The period switcher is a 'use client' component that calls a Server Action to re-fetch metrics for the selected period, updating the cards without a full navigation.

Pros

  • Instant period switching with no page flash
  • Can animate metric value changes
  • Still server-rendered on first load

Cons

  • More complex: client state + server actions + loading states
  • Breaks from every other portal page's pattern
  • Period selection not in URL (loses shareability)
C

Client-Side Fetching with Route Handlers

Build API route handlers (/api/metrics, /api/payouts) and fetch from client components using useEffect or SWR. Full client-side rendering for the data sections.

Pros

  • Maximum interactivity
  • Can poll for real-time updates

Cons

  • Completely breaks existing codebase patterns
  • Loading spinners everywhere on first visit
  • More API surface to maintain and secure
  • Unnecessary complexity for what is essentially a report page
